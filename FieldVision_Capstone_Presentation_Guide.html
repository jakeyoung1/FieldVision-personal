The Problem (Slide Bullets)

The Specific Ask

Build a system that: (1) ingests handwritten or typed scouting notes as PDF/TXT files, (2) produces structured analysis reports benchmarked against historical professional scouting, (3) integrates Trackman pitch data for post-game analytics, and (4) maintains a persistent talent pool across sessions.

Source 1: Branch Rickey Corpus

What it is: 1,919 digitized scouting transcriptions from Branch Rickey's professional career, stored as a CSV knowledge base (data/branch-rickey-scouting.csv).

Why it matters: Rickey is considered the architect of modern baseball scouting — his frameworks for evaluating speed, arm, bat, and character remain foundational. This corpus gives FieldVision's AI historical grounding unavailable anywhere else.

Characteristics: Free-form text, highly domain-specific vocabulary, pre-1960s baseball terminology. Required custom TF-IDF indexing — general-purpose embeddings underperform on this register.

Source 2: Uploaded Scouting Notes

What it is: PDF, TXT, or Markdown files uploaded by coaches or scouts in real time.

Why it matters: Primary user-generated data. Every analysis session revolves around this input.

Challenges: Handwritten PDFs are the hardest case — text extraction libraries (pdfplumber, pypdf) fail on scanned images. Required a Vision OCR fallback using Claude Opus 4.5 to read pages as images. Multi-page PDFs are grouped by player name to build unified player files.

Source 3: Trackman CSV Exports

What it is: Machine-generated CSV files from Trackman ball-tracking hardware, used at college and professional games. Each row is one pitch; key columns include Pitcher, PitcherTeam, PitchType, RelSpeed, SpinRate, InducedVertBreak, HorzBreak, PlateLocHeight, PlateLocSide.

Processing: Per-pitcher aggregations (avg/max velocity, avg spin rate, pitch mix). Team separation extracted via PitcherTeam column. Outputs feed an AI narrative interpretation. Data is multi-team (both home and away pitchers in same file) — neutral, objective framing required.

Data Challenges Summary

  • Handwriting recognition: Scanned PDFs require Vision-model OCR — no standard library handles this.
  • Domain vocabulary: Baseball scouting terminology is narrow and specialized; general NLP models lose nuance.
  • Multi-player documents: Single uploads often contain notes on 3–5 players; automatic player segmentation by name was required.
  • Unstructured → structured: The end goal is turning free-form prose into grade, strengths list, concerns list, and summary — a non-trivial extraction task.

OCR Fallback Chain — Why Three Layers?

  • pdfplumber — Fast, accurate for digitally-generated PDFs. Fails silently on image-only scans.
  • pypdf — Secondary text extractor, handles some edge cases pdfplumber misses. Also fails on handwriting.
  • Claude Opus 4.5 Vision — Final fallback: renders each PDF page as a base64 image, sends to Claude's vision model. Handles handwriting, stamps, diagrams. Only invoked when both text extractors return empty strings.
  • Implementation: backend/services/files.py — three-function cascade with empty-string detection at each stage.

RAG: TF-IDF Retrieval from Branch Rickey Corpus

InputFirst 500 chars of player text
→
VectorizeTF-IDF transform (sklearn, 20K features)
→
RetrieveCosine similarity vs. 1,919 Rickey docs
→
OutputTop-k context block injected into Claude prompt

Why TF-IDF, Not Neural Embeddings?

  • Domain fit: Baseball scouting vocabulary is narrow. Term frequency over a domain-specific corpus outperforms general-purpose sentence embeddings (all-MiniLM, etc.) on exact keyword matching (e.g., "tools," "arm strength," "bat speed").
  • Speed & zero runtime cost: sklearn TF-IDF fits in RAM at startup, returns results in milliseconds. No GPU, no vector database (FAISS), no embedding API calls.
  • Interpretability: TF-IDF weights are explainable — we can show which Rickey document matched and why. Neural similarity scores are opaque.
  • Practical tradeoff: For semantic search over general text, neural embeddings win. For keyword-heavy, domain-specific matching, TF-IDF is competitive and cheaper.
The RAG design choice is a great place to demonstrate technical depth. Acknowledge that neural embeddings (FAISS + all-MiniLM) were considered and evaluated — they were outperformed on this specific domain because baseball scouting language is lexically consistent (the same words appear repeatedly). You chose the right tool for the task rather than the trendy tool.

Claude API Calls — What Each Does

analyze_notes(text, context): Main report generation. Takes raw scouting text + RAG context. Returns structured Markdown: Player Overview, Key Strengths, Areas of Concern, Historical Comparison, Recommendation & Grade. System prompt: SYSTEM_SCOUT (FieldVision scout persona trained on Rickey).

extract_player_profile(label, insights): Parses the Markdown report into a structured JSON object with keys: name, position, grade, strengths[], concerns[], summary. Used to populate the Players tab.

extract_players_from_chat(reply, context): Given a chat reply that mentions players, extracts a JSON array of profiles for any player that is meaningfully evaluated — not just named. Enforces grade differentiation and locks against existing grades from the scouting context.

interpret_pitch_metrics(summary, focus): Translates Trackman stats into plain-language coach narrative. Neutral third-person. System prompt: SYSTEM_COACH.

Grade Scale — Full Plus/Minus

A — Elite prospect A– — Near-elite B+ — Above avg B — Solid B– — Above avg, questions C+ — Avg with upside C — Average (baseline) C– — Below avg, watching D+ — One redeeming tool D — Significant issues F — Not recommended

C is calibrated as the baseline for an average player at this competitive level. B indicates above-average performance. A grades are reserved for elite/standout performers.

What Was Validated

  • Functional correctness: All five API endpoints return expected schemas on valid inputs and appropriate error codes (400, 422, 500) on invalid inputs.
  • OCR accuracy: Vision fallback correctly extracted text from handwritten test PDFs where pdfplumber returned empty strings.
  • Grade differentiation: After prompt engineering, chat-extracted profiles assign different grades to players in the same reply when language implies ranking. (Resolved "everyone gets B" bug.)
  • Grade consistency: Grade lock mechanism prevents a player's grade from changing after their first profile entry unless explicitly requested.
  • Trackman team separation: PitcherTeam grouping correctly separates pitchers from two teams in the same CSV file.

Evaluation Approach & Limitations

  • No ground-truth grades: LLM-assigned grades cannot be objectively measured — evaluation was qualitative, based on coaching staff feedback and face validity.
  • RAG relevance: TF-IDF retrieval was validated by spot-checking retrieved Rickey documents for thematic relevance to input notes — not formally measured with NDCG or MRR.
  • Scale: Tested on 10–15 real player notes. Large-scale load testing not performed on Render free tier.
  • Bias risk: Rickey's documents reflect 1930s–50s scouting norms, including outdated characterizations. Historical context is clearly labeled as reference, not ground truth.

System Capabilities Delivered

  • End-to-end pipeline from raw file upload to structured player profile — no manual data entry.
  • Handwritten PDF scouting notes successfully processed via Claude Vision OCR (novel capability; standard NLP pipelines cannot do this).
  • Persistent Players tab auto-populated across three input channels: direct upload, scouting chat, combined chat.
  • Trackman CSV → per-pitcher stats + AI narrative interpretation, with team-level grouping.
  • Session memory: all uploads accumulate as context for follow-up chat queries.
  • PDF export of all reports (client-side, no server roundtrip).

Slide 9 of 11 Challenges & What We'd Do Differently
SECTION: REFLECTION  |  SUGGESTED TIME: 1.5–2 min

Slide Objective

Demonstrate technical maturity and self-awareness. Every strong project has obstacles — naming them honestly builds credibility with faculty evaluators.

Hard Problems Encountered

  • Grade inflation in LLM output: Initial prompts caused all extracted players to receive a "B" regardless of language. Required careful prompt engineering — explicit baseline calibration (C = average), mandatory grade differentiation, and strict anchoring language tied to specific grades.
  • CSS class name constraints: Grade badges like "B+" can't become CSS class names (invalid characters). Solved with a gradeClass() helper that maps to letter-family color classes.
  • Grade lock consistency: Preventing grade drift across chat turns required a two-level lock: (1) skip players already in the pool on extraction, (2) only update grades when the user explicitly uses re-grade language.
  • Handwritten PDF coverage: Standard libraries fail completely on scanned handwriting. The Vision OCR fallback was essential but added latency and API cost.
  • Render cold starts: Free-tier deployment sleeps after 15 min inactivity. Implemented UptimeRobot keep-alive ping on /api/health.

What We'd Do Differently

  • Formal evaluation dataset: Build a ground-truth set of scouting notes with known player grades, validated by coaching staff, to enable quantitative RAG precision measurement.
  • Database layer: Current state is in-memory (session-scoped). Adding a lightweight database (SQLite → PostgreSQL) would enable cross-session player history, search, and analytics.
  • Fine-tuning vs. prompting: Evaluate whether fine-tuning a smaller model on domain-specific scouting reports would reduce API cost and improve grading consistency over pure prompt engineering.
  • User authentication: Multi-coach support requires user accounts, role-based access (head coach vs. assistant scout), and note ownership.
  • Comprehensive testing: End-to-end integration tests for each API route. Property-based testing for grade extraction edge cases.
The "grade inflation" bug is a great story — walk through it briefly: "Our first extraction prompt produced a 'B' for every single player, regardless of the language used. We realized the model was anchoring to the letter grade examples and treating B as a safe default. We fixed this by rewriting the prompt to: (1) set C as the explicit baseline for an average player, (2) require that players in the same reply receive different grades if the language implies ranking, (3) provide specific linguistic anchors for each grade level." This shows you understand LLM behavior deeply.
Slide 10 of 11 What We Learned
SECTION: REFLECTION  |  SUGGESTED TIME: 1.5 min

Slide Objective

Articulate growth — technical, professional, and methodological. This slide shows the capstone experience had substance beyond just "building a thing."

Key Learnings

  • Prompt engineering is engineering: LLM output quality is as sensitive to prompt design as code is to algorithm design. The difference between "assign a grade" and "C is average; B is above average; differentiate grades when language implies ranking" produced dramatically different results. This is a real engineering discipline.
  • RAG design is domain-specific: The decision between TF-IDF and neural embeddings is not "newer is better." It depends on your corpus vocabulary, query distribution, and computational constraints. Domain knowledge drove a better architectural choice.
  • Production infrastructure matters from Day 1: Designing for Render deployment shaped decisions about statelessness, cold starts, and keep-alive patterns from the beginning. Building for localhost first created technical debt we had to refactor.
  • Stakeholder alignment is ongoing: The April 2 coaching staff meeting revealed requirements (mass upload, player comparison) that weren't in the original spec. Real clients have evolving needs — building extensible architecture rather than hardcoded solutions paid off.
  • LLMs are powerful orchestrators: The most architecturally interesting insight: Claude isn't just generating text. It's functioning as a structured data extraction engine (JSON output), an OCR model (Vision), a retrieval re-ranker (chat grounding), and a reasoning layer — all through a single API with prompt-controlled behavior.
Pick 2–3 of these learnings that feel most personally resonant and expand on them with a brief anecdote. The "prompt engineering is engineering" point is the most original and will land well with a technical audience. Avoid generic statements like "I learned a lot about teamwork" — be specific and technical.
